## 一、简历版本（精炼版，约 5-6 行）

### 推荐写法

> **ScholarMind — 面向通信感知领域的多模态学术研究 Agent**
>
> - 基于 **MCP 协议**构建了 5 个工具服务（论文搜索、PDF 多模态解析、知识图谱、学习路径、代码沙箱），通过 Antigravity Harness 实现 LLM 自主 Workflow 编排
> - 设计并实现了 **Schema-Guided Knowledge Extraction** 管线：定义 7 种节点 + 11 种关系的领域本体，利用 LLM 从论文文本中零标注抽取结构化三元组，入图后自动去重合并
> - 基于 **NetworkX + PageRank** 实现知识图谱分析引擎：综合 PageRank、介数中心性、拓扑排序生成个性化学习路径，支持三种知识盲区检测（基础缺失/孤立概念/单源依赖）
> - 集成 **Vision LLM** 多模态能力：将 PDF 页面渲染为图片直接发送给视觉模型分析，支持公式/框图/性能曲线等学术图表的语义理解
> - 实现**安全代码沙箱**（子进程隔离+超时+危险 import 检测），内置 OFDM/MIMO/MUSIC 等通信仿真模板，支持论文方法的快速复现验证

## 二、面试项目介绍话术（2-3 分钟版本）

### 开场

> 我做了一个面向通信感知领域的学术研究 Agent，叫 ScholarMind。它解决的核心问题是：**研究生读论文效率低，知识积累碎片化，缺乏系统的知识管理和学习路径规划**。

### 架构

> 架构上，我采用了 **MCP 协议 + Harness** 的方式。Antigravity 作为宿主 Agent 提供 LLM 运行时环境，我在它之上构建了 5 个 MCP 工具服务，各自负责不同能力：论文搜索、PDF 解析、知识图谱、学习路径和代码执行。LLM 通过阅读工具的 docstring 自主决定调用哪个工具，不需要我手写编排逻辑。

### 核心亮点

> 最核心的技术是**知识图谱的自动构建**。我定义了一套通信感知领域的知识本体——7 种节点类型、11 种关系类型——然后把这个 Schema 嵌入到抽取 Prompt 中，让 LLM 在 Schema 的约束下输出结构化 JSON。这样做的好处是零标注数据、零训练成本，但能实现领域特定的知识抽取。
>
> 抽取后的知识写入 NetworkX 图，我在上面跑了 PageRank 和介数中心性算法来评估概念重要性，再结合拓扑排序和知识盲区检测，生成个性化的学习路径推荐。

### 多模态

> 另一个亮点是**多模态图表分析**。学术论文中有大量公式、框图、性能曲线，传统 PDF-to-Text 会丢失这些信息。我的方案是把 PDF 页面直接渲染为图片，发送给 Vision API 分析——这样模型能同时理解图和文字的语义关系。

### 工程细节

> 工程方面也有一些考虑。比如论文搜索模块实现了**主动限速 + Exponential Backoff** 的双重保护，避免 API 限流；JSON 输出用了 3 策略级联提取来保证鲁棒性；代码执行模块做了 4 层安全防护。

### 收尾

> 总体来说，这个项目让我对 LLM Agent 的工程落地有了深入理解——从 Prompt 设计到工具编排，从知识图谱构建到安全沙箱，从 API 集成到异常处理。它不仅仅是一个 chatbot，而是一个能真正辅助学术研究的工具型 Agent。

## 三、项目优势与不足的坦诚总结

### ✅ 值得肯定的

| 维度 | 具体 |
|:---|:---|
| **架构设计** | 清晰的分层（MCP Server → 底层模块 → Schema → 数据），模块间解耦 |
| **代码质量** | 极其详细的工程注释和踩坑记录，代码即文档 |
| **防幻觉** | 论文数据来自真实 API + 可验证引用，这是学术 Agent 的底线 |
| **领域定制** | Schema 针对通信感知领域设计，不是通用 Agent |
| **渐进式设计** | Phase 0-4 的演进路径清晰，每个阶段解决一个核心问题 |

### ⚠️ 客观不足

| 维度 | 问题 | 严重程度 |
|:---|:---|:---|
| 无 RAG/向量检索 | ChromaDB 在依赖中但未使用 | 中 |
| async/sync 混用 | extractor 中同步 LLM 调用阻塞事件循环 | 中 |
| 无系统化评测 | 缺少自动化测试和 benchmark | 中 |
| 实体标准化弱 | 只有大小写标准化，无同义词处理 | 中 |
| 无状态恢复 | Agent 中断后对话上下文丢失 | 低 |
| JSON 提取器重复 | multimodal.py 和 extractor.py 各有一份 | 低 |

## 四、"写厚再写薄"——核心凝练

整个项目可以凝练为**三句话**：

1. **工具层**：通过 MCP 协议构建 5 个学术工具服务，实现论文搜索→解析→知识抽取→路径规划→代码验证的全链路闭环
2. **知识层**：Schema-Guided LLM Extraction + NetworkX 图分析（PageRank + 盲区检测 + 拓扑排序），构建领域知识图谱并生成学习路径
3. **感知层**：Vision LLM 直接理解 PDF 页面（公式/框图/曲线），突破传统 PDF-to-Text 的信息损失瓶颈

这三层构成了 ScholarMind 的核心竞争力，也是面试时最值得展开的三个 Key Point。

## 五、吸引面试官的 Key Points

1. **"我没有用 LangGraph，而是用了 Harness 模式"** → 展示你对 Agent 架构的独立思考
2. **"Schema-Guided Extraction 实现了零标注的领域知识抽取"** → 展示 LLM 工程化能力
3. **"PageRank 在学术知识图谱中的含义是..."** → 展示算法理解深度
4. **"JSON 输出用了 3 策略 fallback"** → 展示生产级工程意识
5. **"主动限速 vs 被动 Backoff"** → 展示 API 集成经验
