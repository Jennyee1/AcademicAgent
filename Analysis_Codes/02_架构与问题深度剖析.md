# ScholarMind 架构与问题深度剖析

## 一、Agent 架构模式分析

### 1. 采用了什么模式？

ScholarMind **不是**经典的 ReAct 模式，而是 **MCP Tool-Use + Harness 工程模式**。

#### 准确定义

```
Antigravity (宿主 LLM / Harness)
  ├── GEMINI.md (角色 + 领域规则 + 术语表)
  ├── Skills (paper_reader, learning_path) → 静态 Prompt 模板
  ├── Workflows (paper-analysis, knowledge-build, simulation) → 预定义步骤
  └── MCP Servers (5个独立进程)
      ├── paper_search   → Semantic Scholar / arXiv API
      ├── paper_reader    → PDF 解析 + Vision 分析
      ├── knowledge_graph → LLM 知识抽取 + NetworkX 存储
      ├── learning_path   → PageRank 图分析
      └── code_execution  → 安全沙箱执行
```

#### 与 ReAct 的区别

| 维度 | ReAct | ScholarMind |
|:---|:---|:---|
| 循环方式 | Thought → Action → Observation 显式循环 | LLM 自主决定是否调用工具 |
| 工具调用 | Agent 代码中的 while 循环 | Antigravity Harness 内置循环 |
| 状态管理 | Agent 自己维护 | 交给 Harness（Antigravity） |
| 代码量 | 需要写循环/状态机 | 零循环代码，只写工具定义 |

#### 为什么选择这种模式？

1. **零编排代码**：不需要自己写 LangGraph 状态机、ReAct 循环——Antigravity 作为 Harness 已经内置了 tool-use 循环
2. **LLM 自主决策**：LLM 根据 docstring 自主决定调用哪个工具，不需要硬编码路由规则
3. **快速迭代**：新增功能只需添加一个 `@mcp.tool()` 函数，不需要修改编排逻辑

### 2. 关于 "Model Routing" 描述的纠正

> 原描述："整体采用了 Model Routing 策略，将任务分解交给不同的模型处理"

**这个描述不准确**。准确说法：

- ScholarMind 采用的是 **Tool Routing**（工具路由），不是 Model Routing（模型路由）
- LLM 是同一个模型（MiniMax / Claude），但通过 MCP 协议分发到不同的**工具服务器**
- Antigravity 作为 **Harness**，让 LLM 自主决策调用哪个工具

> 正确描述：利用 Antigravity 提供的 Harness 运行环境，构建了一个通过 MCP 协议暴露多种学术工具的智能体系统。LLM 在 Harness 的 tool-use 循环中自主决策 workflow，按需调用论文搜索、知识抽取、图表分析、代码执行等工具。

### 3. LangGraph vs Harness

ScholarMind **没有使用 LangGraph**。

| 维度 | LangGraph 方式 | Harness (Antigravity) 方式 |
|:---|:---|:---|
| 流程定义 | 代码定义节点图 + 条件边 | 自然语言 Workflow + LLM 自主决策 |
| 优势 | 精确控制、可调试、可复现 | 零编排代码、灵活适应、快速迭代 |
| 劣势 | 需要大量样板代码 | LLM 决策不确定性、难以调试 |
| 适合场景 | 生产级、复杂业务流程 | 原型验证、研究探索 |

## 二、工具层详解

### MCP 工具设计三原则

1. **docstring 驱动路由**：LLM 通过 docstring 判断何时调用
2. **Markdown 输出**：LLM 可直接展示给用户
3. **友好错误信息**：LLM 据此调整策略

### Skill 的作用

Skill 是**静态的指令模板**，告诉 Antigravity："当用户说 X 时，执行这些 shell 命令"。

```markdown
# SKILL.md
## When to Use
Use this skill when the user:
- Provides a PDF paper and asks to "analyze", "read", or "summarize" it

## Instructions
### 1. Extract Paper Structure
cd AcademicAgent && python skills/paper_reader/scripts/parse_pdf.py --action structure --pdf "<path>"
```

**Skill vs MCP Tool 的关系**：
- Skill = 给 LLM 的"说明书"，教它如何组合底层命令
- MCP Tool = 标准化的函数接口，LLM 直接调用
- 二者互补：Skill 处理多步骤流程，MCP Tool 处理单步操作

## 三、多模态解析能力

### Vision API 方案 vs 传统 OCR

ScholarMind 选择 **Vision LLM 直接理解 PDF 页面**，而非依赖 PDF-to-Text。

**为什么？**

| 维度 | PDF-to-Text | Vision LLM |
|:---|:---|:---|
| 公式 | `\sqrt{x}` 变成乱码 | 直接理解 LaTeX 语义 |
| 图表 | 丢失所有视觉信息 | 理解框图/曲线/表格 |
| 双栏排版 | 两栏文字混排 | 理解版面布局 |
| 扫描版 PDF | 完全失败 | 正常工作 |

**技术实现**：
```python
# 整页渲染为 PNG → base64 → 发送给 Vision API
page_b64 = parser.render_page_as_image(page_num, dpi=200)
data_uri = f"data:image/png;base64,{page_b64}"
response = client.chat.completions.create(
    model=self.model,
    messages=[{"role": "user", "content": [
        {"type": "image_url", "image_url": {"url": data_uri}},
        {"type": "text", "text": prompt},
    ]}],
)
```

### 局限性

1. **Token 成本高**：一张 200 DPI 的 A4 页面约消耗 1000-2000 vision tokens
2. **不支持批量**：逐页分析，30 页论文需要 30 次 API 调用
3. **数据提取精度**：Vision LLM 读取图表数据点不如专业 OCR 精确

## 四、上下文与记忆管理

### 当前实现

ScholarMind 的记忆机制分两层：

| 层级 | 实现 | 持久化 |
|:---|:---|:---|
| 短期记忆 | Antigravity 对话上下文窗口 | 会话内 |
| 长期记忆 | 知识图谱 JSON 文件 | 磁盘持久化 |

### 长文本处理策略

1. **分段抽取**：超过 3000 字符自动切割，按段落边界分段
2. **章节分割**：正则识别论文章节，按 section 单独处理
3. **本地合并**：多段抽取结果按 node_id 去重合并

### 未解决的问题

**"Lost-in-the-Middle" 问题**：
- 当论文超过 8000 token 时，LLM 对中间部分的注意力会下降
- 当前方案（分段）部分缓解了这个问题，但可能丢失跨段落的关联信息

**改进方案**：
1. **分层摘要（Summarize-Map-Reduce）**：先对每段生成摘要，再对摘要做全局分析
2. **Sliding Window + Overlap**：分段时保留 20% 重叠，避免截断语义

## 五、知识图谱构建深度分析

### 技术实现

```
论文文本 → LLM Schema-Guided Extraction → KGNode + KGEdge
         → KnowledgeGraphStore (NetworkX DiGraph)
         → JSON 持久化 → PageRank 图分析
```

### 回应质疑

**Q1：实体/关系抽取质量如何保证？**

当前方案：Schema-Guided Prompt + 3 策略 JSON 提取
- **优势**：零标注数据，LLM 天然理解领域术语
- **劣势**：LLM 抽取不稳定（"LLM" vs "大语言模型" vs "Large Language Model" 可能生成三个孤立节点）
- **缓解措施**：`_normalize_label()` 做了基础的大小写标准化，但**不够**

**改进建议**：
1. 在 Prompt 中嵌入**标准学术词汇表**（如 `{"OFDM": ["正交频分复用", "Orthogonal Frequency Division Multiplexing"]}`）
2. 入库后跑**聚类算法**：用 Embedding 相似度批量合并相似节点
3. 维护**实体别名映射表**（alias table）

**Q2：为什么不用 GraphRAG / 向量数据库？**

当前状态：requirements.txt 包含了 `chromadb>=0.5.0`，但代码中**未实际使用**。

**为什么现在不用？**
1. **MVP 优先**：<100 篇论文，NetworkX + JSON 够用
2. **向量检索的价值在"模糊查询"**——当前场景下关键词搜索已满足需求
3. **GraphRAG 增加了工程复杂度**（需要维护向量索引 + 图结构索引双系统）

**应该考虑的方向**：
```
混合检索路由：
用户查询 → 意图分类
  ├── 精确查询 ("OFDM 和什么方法相关？") → 图谱子图匹配
  ├── 模糊查询 ("类似注意力机制处理图表的文章？") → 向量相似度检索
  └── 复合查询 → GraphRAG (向量召回 + 子图匹配 + LLM 综合)
```

**Q3：知识图谱 Schema 维护成本？**

当前 Schema 有 7 种节点 + 11 种关系类型，针对通信感知领域定制。
- **优势**：Schema 约束让 LLM 输出规范化
- **风险**：新领域需要扩展 Schema——但由于使用 Enum，只需追加枚举值

## 六、JSON 格式稳定性

### 当前方案：3 策略 Fallback

```python
# 策略 1: 直接 json.loads(text)
# 策略 2: 提取 ```json ``` 代码块
# 策略 3: 找最外层 { } 对
```

### 改进建议

1. **Structured Output API**：OpenAI / MiniMax 支持 `response_format={"type": "json_object"}`——但当前代码未使用
2. **Pydantic Validation**：用 Pydantic 模型验证 JSON 结构完整性
3. **重试机制**：JSON 解析失败时自动重新请求（当前缺失）

## 七、模型幻觉处理

### 当前措施

| 措施 | 实现位置 |
|:---|:---|
| 真实 API 数据源 | paper_search.py — 论文来自 Semantic Scholar |
| 可验证引用 | DOI / arXiv ID 附带在搜索结果中 |
| Schema 约束 | extractor.py — 只允许 Schema 中定义的类型 |
| Prompt 明确指令 | "Do NOT hallucinate entities not supported by the text" |

### 不足

1. **无事后验证**：LLM 抽取的实体/关系没有交叉验证机制
2. **无引用溯源**：知识图谱中的三元组无法追溯到原文具体位置
3. **无用户反馈循环**：用户无法标记"这个抽取结果是错的"

## 八、死循环 / 异常处理 / 状态恢复

### 循环控制
- Agent 的 tool-use 循环由 Antigravity Harness 管理，**代码层面无自定义循环**
- Harness 自带最大轮次限制（通常 10-25 轮）

### API 异常处理
- paper_search：主动限速 + Exponential Backoff + 友好错误信息
- 其他 MCP Server：try-except 捕获所有异常，返回格式化错误信息

### 状态恢复
- **当前无状态恢复机制**——如果 Agent 中断，知识图谱保留（已持久化），但对话上下文丢失
- **改进建议**：添加 checkpoint 机制，定期保存抽取进度

### 安全管控
- 代码沙箱：4 层安全防护（子进程隔离 + 超时 + 目录隔离 + 危险 import 检测）
- API Key：环境变量管理，不硬编码
- 成本控制：GEMINI.md 中规定"单次任务最多搜索 3 轮，每轮最多 5 篇"

## 九、评测与 BadCase 挖掘

### 当前状态：无系统化评测

项目缺少：
1. 自动化测试管线（tests/ 有文件但无 CI）
2. 端到端评测基准
3. BadCase 收集和分析机制

### 建议的评测维度

| 指标 | 定义 | 衡量方式 |
|:---|:---|:---|
| 工具调用准确率 | Agent 选对工具的比例 | 人工标注 20 个 case |
| 知识抽取 F1 | 实体/关系抽取的精确率和召回率 | 对比人工标注的 ground truth |
| 学习路径相关性 | 推荐的学习路径是否符合领域逻辑 | 领域专家评审 |
| 端到端完成率 | 用户请求被完整解决的比例 | 用户满意度调查 |
