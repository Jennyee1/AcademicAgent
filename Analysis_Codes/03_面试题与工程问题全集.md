# ScholarMind 面试八股题与工程问题全集

---

## 一、Agent 架构类

### Q1: 你的 Agent 采取了什么架构模式？

**答**：ScholarMind 采用 **MCP Tool-Use + Harness** 架构，而非经典 ReAct。

- Antigravity 作为 Harness，提供 LLM 运行时环境和内置的 tool-use 循环
- 5 个 MCP Server 作为独立进程，通过 stdio 与 LLM 通信
- LLM 根据工具的 docstring 自主决策调用哪个工具
- 不需要手写 ReAct 循环或 LangGraph 状态机

**追问：为什么不用 LangGraph？**

LangGraph 适合需要精确控制流程的生产级系统。ScholarMind 是学术研究场景的 MVP，使用 Harness 方式可以：
1. 零编排代码——只写工具定义
2. LLM 自主决策 workflow——更灵活
3. 快速迭代——新增功能只需 `@mcp.tool()`

**追问：那你怎么防止死循环？**

Harness（Antigravity）自带最大轮次限制。此外，GEMINI.md 中定义了成本控制规则："单次任务最多搜索 3 轮，每轮最多 5 篇"——从 Prompt 层面约束 LLM 行为。

---

### Q2: MCP 协议是什么？为什么用 MCP？

**答**：MCP (Model Context Protocol) 是 Anthropic 推出的标准化协议，定义了 LLM 与外部工具之间的通信格式。

**核心优势**：
1. **进程隔离**：每个 MCP Server 独立进程，崩溃不影响主 Agent
2. **标准化接口**：`@mcp.tool()` 装饰器自动生成 JSON Schema，LLM 自动发现工具
3. **可组合性**：同一个 MCP Server 可以被多个 LLM 客户端共享
4. **热插拔**：新增/移除工具不需要重启 Agent

**与 Function Calling 的区别**：
Function Calling 是 LLM API 层面的特性（函数定义在同一进程中），MCP 是进程间通信协议（工具在独立进程中）。

---

### Q3: 你的工具是怎么被 LLM "发现" 和 "选择" 的？

**答**：通过 **docstring 驱动路由**。

```python
@mcp.tool()
async def search_papers(query: str, limit: int = 5) -> str:
    """
    通过 Semantic Scholar API 搜索学术论文。
    
    适合使用的场景：
    - 用户说"帮我找关于xxx的论文"
    
    不适合使用的场景：
    - 用户已经提供了论文 PDF
    """
```

LLM 读取所有工具的 docstring 和参数 schema，自主判断当前用户请求应该调用哪个工具。

## 二、知识图谱类

### Q4: 知识图谱是怎么构建的？

**答**：**Schema-Guided LLM Extraction**。

1. 定义领域 Schema：7 种节点类型（Paper, Author, Concept, Method, Dataset, Metric, Tool）+ 11 种关系类型
2. 将 Schema + JSON 示例嵌入 Extraction Prompt
3. LLM 被约束为只能输出 Schema 中定义的类型
4. 3 策略 JSON 提取器解析 LLM 输出
5. 写入 NetworkX DiGraph，自动去重合并

**追问：为什么不用传统 NER？**

传统 NER 需要大量标注数据，且不理解通信感知领域术语。LLM 天然理解 "OFDM"、"Beamforming"、"CRLB"。Schema-guided prompt = 零标注数据的领域知识抽取。

### Q5: LLM 抽取实体名不一致怎么办？（"LLM" vs "大语言模型"）

**答**：当前方案：`_normalize_label()` 做了基础的大小写标准化（`"OFDM" == "ofdm"`）。

**不足**：无法处理 "OFDM" vs "正交频分复用" 的跨语言同义词问题。

**改进方案**：
1. 在 Prompt 中嵌入预定义的**标准学术词汇表**
2. 入库后定期跑 Embedding 相似度**聚类合并**
3. 维护**实体别名映射表**

### Q6: 为什么用 NetworkX + JSON，不用 Neo4j？

**答**：MVP 阶段的务实选择。

| 维度 | NetworkX + JSON | Neo4j |
|:---|:---|:---|
| 部署成本 | 零依赖 | 需要数据库服务 |
| 数据规模 | <100 篇论文足够 | 1000+ 篇论文 |
| 查询性能 | O(n) 遍历 | 原生图查询 O(1) |
| 学习成本 | Python 原生 | 需要学 Cypher |

API 设计对标 Neo4j，迁移成本低。

### Q7: 为什么不用 GraphRAG / 向量数据库？

**答**：当前未使用（虽然 requirements.txt 包含 chromadb），原因是 MVP 优先。

**应该使用的场景**：
- 模糊查询（"有没有类似注意力机制的文章？"）→ 需要向量检索
- 跨论文综合分析 → 需要 GraphRAG 的子图匹配 + 向量召回

**微软 GraphRAG 方案分析**：
先用 LLM 从非结构化文档中抽取实体/关系构建图谱，同时保留向量索引。查询时结合向量召回和子图匹配——这是ScholarMind的自然演进方向。

## 三、多模态 / 长文本类

### Q8: 如何处理学术 PDF 中的公式和图表？

**答**：**Vision LLM 直接理解 PDF 页面**。

将 PDF 页面渲染为 200 DPI PNG 图片 → base64 编码 → 发送给 Vision API + 领域特定的分析 Prompt。

**优势**：理解公式语义（而非 OCR 为乱码），理解框图/曲线的含义，处理扫描版 PDF。

**不足**：成本高（每页 1000-2000 vision tokens），不支持批量分析。

### Q9: 超长论文怎么处理？（token 溢出 / 注意力失焦）

**答**：三重策略。

1. **章节分割**：正则识别论文结构（Abstract / Method / Experiments），按章节独立处理
2. **分段抽取**：超过 3000 字符自动按段落边界切割，各段独立抽取后本地合并
3. **短期 + 长期记忆分离**：当前对话用上下文窗口（短期），知识图谱 JSON 持久化（长期）

**追问：Lost-in-the-Middle 问题怎么解决？**

当前的分段策略部分缓解了这个问题。进一步改进：
- Sliding Window + 20% 重叠
- 分层摘要（Map-Reduce）：先段落摘要，再全局综合

### Q10: 你的短期记忆和长期记忆是怎么联动的？

**答**：
- **短期**：Antigravity 对话窗口（会话内有效）
- **长期**：知识图谱 JSON 文件（跨会话持久化）
- **联动方式**：当用户问"这个概念和什么相关？"时，LLM 调用 `query_knowledge` MCP 工具查询长期记忆（知识图谱），结果注入当前对话上下文

## 四、JSON 输出 / 鲁棒性类

### Q11: 如何保证 LLM 输出 100% 合法 JSON？

**答**：**不可能 100%**——必须有 fallback。

当前方案：3 策略级联提取
1. 直接 `json.loads()`
2. 提取 ```json``` 代码块
3. 找最外层 `{ }` 对

**改进方向**：
1. 使用 API 的 `response_format={"type": "json_object"}`（结构化输出）
2. Pydantic 模型验证 JSON 完整性
3. JSON 解析失败时自动重试

### Q12: API 出现波动怎么办？

**答**：
- paper_search：**主动限速 + Exponential Backoff + 最大 3 次重试**
- 所有 MCP Server：try-except 兜底，返回**人类可读的错误信息**，LLM 据此调整策略
- **不足**：知识抽取模块缺少重试机制（应补充）

## 五、安全 / 评测类

### Q13: 代码沙箱的安全机制？

**答**：4 层防护。

1. **子进程隔离**：`subprocess.run()`，崩溃不影响主进程
2. **超时控制**：默认 30s，上限 120s
3. **工作目录隔离**：每次执行在独立 tempdir
4. **危险 import 检测**：正则黑名单（os.system, subprocess, eval, exec）

**追问：为什么不用 Docker？**

部署复杂度高（学生需要装 Docker Desktop），当前安全级别对学术场景够用。保留 Docker 接口，Phase 5 可升级。

### Q14: 如何评测 Agent 性能？

**答**：当前无系统化评测（这是不足）。建议的评测维度：

| 指标 | 衡量方式 |
|:---|:---|
| 工具调用准确率 | 人工标注 20+ case |
| 知识抽取 F1 | 对比人工 ground truth |
| 学习路径相关性 | 领域专家评审 |
| 端到端完成率 | 用户满意度 |

### Q15: 幻觉怎么防？

**答**：
1. **真实数据源**：论文搜索来自 Semantic Scholar / arXiv，不让 LLM 编造
2. **可验证引用**：每篇论文附带 DOI / arXiv ID
3. **Schema 约束**：只允许 Schema 中定义的实体/关系类型
4. **Prompt 指令**：明确要求 "Do NOT hallucinate"

## 六、前沿延伸类

### Q16: 对比 Claude Code，你的 Agent 差距在哪里？

| 维度 | Claude Code | ScholarMind |
|:---|:---|:---|
| 上下文理解 | 200K 窗口 + 自主文件搜索 | 依赖分段处理 |
| 代码执行 | 原生 shell 执行 | subprocess 沙箱 |
| 状态恢复 | 自动 checkpoint | 无 |
| 知识积累 | KBAI + 持久化记忆 | 知识图谱 JSON |

**核心差距**：Claude Code 有**自主探索能力**（自动搜索文件、理解代码库），ScholarMind 依赖预定义工具。

### Q17: 如果让你重新设计，会怎么做？

1. **替换 NetworkX 为 LlamaIndex GraphStore**：自带 GraphRAG 支持
2. **添加向量检索层**：ChromaDB 实现模糊查询
3. **实现流式知识抽取**：边读 PDF 边抽取，实时更新图谱
4. **添加用户反馈回路**：让用户标记错误抽取，持续改进

## 七、关于文献对比的标准化维度

在实现文献对比功能时，建议提取以下维度：

| 维度 | 描述 | 重要性 |
|:---|:---|:---|
| **Problem Formulation** | 问题建模方式（优化目标、约束条件） | ⭐⭐⭐⭐⭐ |
| **Signal Model** | 信号模型（OFDM/MIMO 参数、天线配置） | ⭐⭐⭐⭐⭐ |
| **Algorithm** | 算法名称、复杂度（O(N²) / O(NlogN)） | ⭐⭐⭐⭐⭐ |
| **Dataset** | 使用的仿真/实测数据集 | ⭐⭐⭐⭐ |
| **Metrics** | 评估指标（BER, RMSE, SE, 运行时间） | ⭐⭐⭐⭐ |
| **Baseline** | 对比的基线方法 | ⭐⭐⭐⭐ |
| **Performance** | 关键性能数值 | ⭐⭐⭐ |
| **Assumptions** | 关键假设（perfect CSI, narrowband） | ⭐⭐⭐ |
| **Limitations** | 作者承认的局限性 | ⭐⭐⭐ |
