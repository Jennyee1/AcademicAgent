## 关键架构决策记录 (ADR)

### ADR-001: 从单Agent起步，渐进演进到多Agent

- **决策**: Phase 0-2 使用单 Agent（Claude Code + MCP Tools），Phase 3+ 按需引入 sub-agent
- **理由**: 
  - 单 Agent 开发速度快、调试简单
  - MCP 架构天然支持渐进式扩展
  - 可以在 MCP Tool 内部启动 sub-agent 实现隔离
- **权衡**: 
  - 放弃了多 Agent 架构的"上下文隔离"优势
  - 换来了更快的迭代速度和更低的 debug 成本

### ADR-002: Phase 顺序调整（学习规划先于代码执行）

- **决策**: Phase 3 = 学习路径规划，Phase 4 = 代码执行
- **理由**:
  - 学习规划直接承接 Phase 2 的知识图谱，技术连贯性强
  - 代码执行需要 Docker 沙箱，工程难度更高，独立性强
  - 学习规划的用户感知价值高，可以尽早"自己用起来"

### ADR-003: 轻量级知识图谱存储方案

- **决策**: MVP 阶段使用 NetworkX + JSON，不用 Neo4j
- **理由**:
  - 个人使用场景论文量 < 100 篇，NetworkX 完全够用
  - 无需额外部署数据库服务
  - Schema 和数据格式可以后期平滑迁移到 Neo4j

### ADR-004: 论文来源三层体系

- **决策**: 支持在线搜索（Semantic Scholar/arXiv）+ 本地 PDF + Zotero 导入
- **理由**:
  - 在线搜索保证论文真实性（API 返回，非 LLM 编造）
  - 本地 PDF 是硕博生最常用的论文管理方式
  - Zotero 是学术圈主流文献管理器，后期集成增加黏性

### ADR-005: 使用 specSen_torch 环境而非新建 conda 环境

- **决策**: 复用现有 `specSen_torch` (Python 3.10) 作为开发环境
- **理由**:
  - base 环境 Python 3.9 有 SSL 模块 bug（`check_hostname requires server_hostname`）
  - conda create 在校园网代理下失败（SSL + Proxy 双重问题）
  - specSen_torch 的 Python 3.10 没有该 SSL bug
- **风险**: 与现有科研包可能产生依赖冲突 → 后续监控
- **教训**: **环境问题是真实项目中最常被低估的时间杀手**

**遇到的真实 Bug 与修复**:

#### Bug #1: requirements.txt 中文编码炸裂
- **现象**: `pip install -r requirements.txt` 报错 `UnicodeDecodeError: 'gbk' codec can't decode`
- **根因**: Windows 默认编码是 GBK，文件中有中文注释，pip 用 GBK 解码失败
- **修复**: 移除中文注释，只用 ASCII
- **教训**: **在 Windows 环境中，所有会被工具链读取的配置文件（requirements.txt、setup.cfg、Dockerfile）一律只用 ASCII 注释。** 中文注释只放在 .md 文档和 Python 源码（UTF-8 声明）中。

#### Bug #2: Python 3.9 SSL 模块 Bug
- **现象**: `pip install` 和 `conda create` 全部失败，报 `ValueError: check_hostname requires server_hostname`
- **根因**: Python 3.9.7 的 ssl 模块实现有缺陷，与某些代理配置不兼容
- **修复**: 使用 Python 3.10 (specSen_torch 环境)
- **教训**: **base conda 环境不要用来做开发。** 它的 Python 版本通常偏旧，且不能轻易升级。

#### Bug #3: MCP SDK API 版本漂移（真实 Silent Regression）
- **现象**: `FastMCP.__init__() got an unexpected keyword argument 'description'`
- **根因**: MCP SDK v1.27 把构造函数参数从 `description` 改为 `instructions`，无迁移提示
- **修复**: 改用 `instructions` 参数
- **教训**: **这就是文档 05 中描述的 Silent Regression 的真实案例。** 依赖库升级后接口变化但无告警。防御措施：① 锁定依赖版本 ② 写回归测试 ③ 阅读 changelog

#### Bug #4: arXiv API HTTP→HTTPS 迁移
- **现象**: arXiv 搜索返回 `301 Moved Permanently`
- **根因**: arXiv 已将 API 从 `http://export.arxiv.org` 迁移到 `https://export.arxiv.org`
- **修复**: URL 改为 HTTPS + httpx 添加 `follow_redirects=True`
- **教训**: **外部 API 随时可能迁移域名/协议。** 集成测试不仅测功能，也是在监控 API 健康度。`follow_redirects=True` 应该作为 httpx 的默认配置。

#### Bug #5: Semantic Scholar / arXiv 双重限流 (429)
- **现象**: 连续测试后 API 返回 429 Too Many Requests
- **根因**: 无 API Key 时限额极低（Semantic Scholar ~100 req/5min）
- **影响**: 集成测试在限流期间不可靠
- **待修复**: ① 获取 Semantic Scholar API Key ② 测试中加入 retry + backoff ③ 添加请求间隔
- **教训**: **集成测试不要连续跑太快。** 生产环境中需要实现 exponential backoff 和请求队列。

**遇到的真实 Bug 与修复**:

#### Bug #7: pytest.ini GBK 编码错误（复发）
- **现象**: pytest 无法解析 pytest.ini 文件，报 `UnicodeDecodeError: 'gbk'`
- **根因**: pytest.ini 中的中文注释在 Windows GBK 环境下无法正确解码
- **修复**: 将所有中文注释替换为英文注释
- **教训**: **Windows 环境下所有配置文件（.ini, .cfg, .toml）避免使用中文字符**

#### Bug #8: Python 3.9 类型标注不兼容
- **现象**: `str | Path | None` 语法报 `TypeError: unsupported operand type(s) for |`
- **根因**: PEP 604 的 `X | Y` 联合类型语法需要 Python 3.10+
- **修复**: 在所有模块顶部添加 `from __future__ import annotations`
- **教训**: **使用 specSen_torch 环境时（Python 3.9），写类型标注要加 future import**

#### Bug #9: __init__.py 急切导入导致测试失败
- **现象**: 测试无法导入 schema/graph_store，因为 __init__.py 拉入了 extractor（依赖 anthropic SDK）
- **根因**: `__init__.py` 中的 `from .extractor import ...` 在 import 时就执行了
- **修复**: 从 __init__.py 中移除 extractor 的导入，使用时直接 `from src.knowledge.extractor import ...`
- **教训**: **有重型依赖的模块不要在 __init__.py 中急切导入**
