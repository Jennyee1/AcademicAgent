## 关键架构决策记录 (ADR)

### ADR-001: 从单Agent起步，渐进演进到多Agent

- **决策**: Phase 0-2 使用单 Agent（Claude Code + MCP Tools），Phase 3+ 按需引入 sub-agent
- **理由**: 
  - 单 Agent 开发速度快、调试简单
  - MCP 架构天然支持渐进式扩展
  - 可以在 MCP Tool 内部启动 sub-agent 实现隔离
- **权衡**: 
  - 放弃了多 Agent 架构的"上下文隔离"优势
  - 换来了更快的迭代速度和更低的 debug 成本

### ADR-002: Phase 顺序调整（学习规划先于代码执行）

- **决策**: Phase 3 = 学习路径规划，Phase 4 = 代码执行
- **理由**:
  - 学习规划直接承接 Phase 2 的知识图谱，技术连贯性强
  - 代码执行需要 Docker 沙箱，工程难度更高，独立性强
  - 学习规划的用户感知价值高，可以尽早"自己用起来"

### ADR-003: 轻量级知识图谱存储方案

- **决策**: MVP 阶段使用 NetworkX + JSON，不用 Neo4j
- **理由**:
  - 个人使用场景论文量 < 100 篇，NetworkX 完全够用
  - 无需额外部署数据库服务
  - Schema 和数据格式可以后期平滑迁移到 Neo4j

### ADR-004: 论文来源三层体系

- **决策**: 支持在线搜索（Semantic Scholar/arXiv）+ 本地 PDF + Zotero 导入
- **理由**:
  - 在线搜索保证论文真实性（API 返回，非 LLM 编造）
  - 本地 PDF 是硕博生最常用的论文管理方式
  - Zotero 是学术圈主流文献管理器，后期集成增加黏性

### ADR-005: 使用 specSen_torch 环境而非新建 conda 环境

- **决策**: 复用现有 `specSen_torch` (Python 3.10) 作为开发环境
- **理由**:
  - base 环境 Python 3.9 有 SSL 模块 bug（`check_hostname requires server_hostname`）
  - conda create 在校园网代理下失败（SSL + Proxy 双重问题）
  - specSen_torch 的 Python 3.10 没有该 SSL bug
- **风险**: 与现有科研包可能产生依赖冲突 → 后续监控
- **教训**: **环境问题是真实项目中最常被低估的时间杀手**

- **现象**: `pip install` 和 `conda create` 全部失败，报 `ValueError: check_hostname requires server_hostname`
- **根因**: Python 3.9.7 的 ssl 模块实现有缺陷，与某些代理配置不兼容
- **修复**: 使用 Python 3.10 (specSen_torch 环境)
- **教训**: **base conda 环境不要用来做开发。** 它的 Python 版本通常偏旧，且不能轻易升级。

#### Bug #3: MCP SDK API 版本漂移（真实 Silent Regression）
- **现象**: `FastMCP.__init__() got an unexpected keyword argument 'description'`
- **根因**: MCP SDK v1.27 把构造函数参数从 `description` 改为 `instructions`，无迁移提示
- **修复**: 改用 `instructions` 参数
- **教训**: **这就是文档 05 中描述的 Silent Regression 的真实案例。** 依赖库升级后接口变化但无告警。防御措施：① 锁定依赖版本 ② 写回归测试 ③ 阅读 changelog

#### Bug #4: arXiv API HTTP→HTTPS 迁移
- **现象**: arXiv 搜索返回 `301 Moved Permanently`
- **根因**: arXiv 已将 API 从 `http://export.arxiv.org` 迁移到 `https://export.arxiv.org`
- **修复**: URL 改为 HTTPS + httpx 添加 `follow_redirects=True`
- **教训**: **外部 API 随时可能迁移域名/协议。** 集成测试不仅测功能，也是在监控 API 健康度。`follow_redirects=True` 应该作为 httpx 的默认配置。

#### Bug #5: Semantic Scholar / arXiv 双重限流 (429)
- **现象**: 连续测试后 API 返回 429 Too Many Requests
- **根因**: 无 API Key 时限额极低（Semantic Scholar ~100 req/5min）
- **影响**: 集成测试在限流期间不可靠
- **待修复**: ① 获取 Semantic Scholar API Key ② 测试中加入 retry + backoff ③ 添加请求间隔
- **教训**: **集成测试不要连续跑太快。** 生产环境中需要实现 exponential backoff 和请求队列。

**核心算法详解**:

#### 1. 概念重要性评分公式

```
importance = 0.4 × PageRank_norm + 0.3 × Degree_norm + 0.2 × InDegree_norm + 0.1 × Betweenness_norm
```

- **PageRank (0.4)**: 权重最高，因为学术图谱的核心是"被引用的重要性传递"
- **Degree (0.3)**: 直接连接数反映知识的关联密度
- **InDegree (0.2)**: 被引用次数反映知识的被依赖程度
- **Betweenness (0.1)**: 介数中心性低权重，识别"知识桥梁"节点

为什么不用 eigenvector centrality？
→ PageRank 是 eigenvector centrality 的改进版（加了阻尼因子避免 sink 问题），在有向图中更稳定。

#### 2. 知识盲区三种类型

| 类型 | 检测逻辑 | 含义 |
|:---|:---|:---|
| foundation_gap | PageRank 前 30% 但属性 < 2 | 核心概念掌握不够 |
| isolated_concept | 度 ≤ 1 且非论文/作者 | 碎片化知识 |
| single_source | 只从 1 篇论文了解 | 理解可能有偏差 |

为什么盲区类型要分三种？
→ 不同盲区需要不同的补救策略：
  - foundation_gap → 读教材/综述
  - isolated_concept → 探索关系
  - single_source → 交叉验证

#### 3. 学习路径综合排序

```
priority = importance × 0.4 + gap_severity × 0.4 + topo_boost × 0.2
```

- **importance (0.4)**: 越重要的概念越优先学
- **gap_severity (0.4)**: 越是盲区越需要优先补
- **topo_boost (0.2)**: 拓扑排序低位 = 前置知识，先学

为什么用 SCC 凝缩（condensation）处理环？
→ 知识图谱可能存在 A uses B, B uses A 的互相引用。
  nx.condensation() 把强连通分量缩成超节点，保证 DAG，才能拓扑排序。

#### 4. 图谱健康度评估的 5 级判定

```python
if n == 0:           → "⬜ 空图谱"
elif n < 10:         → "🟡 起步阶段"
elif avg_deg < 2:    → "🟡 知识稀疏"
elif components > 30%: → "🟡 知识碎片化"
elif n >= 50 && avg >= 3: → "🟢 健康"
else:                → "🟢 良好"
```

#### Bug #10: 可选依赖导致全量测试无法收集

- **现象**: `python -m pytest tests/` 报 2 个 collection error
  - `test_multimodal.py` → `ModuleNotFoundError: No module named 'anthropic'`
  - `test_paper_search.py` → `ModuleNotFoundError: No module named 'httpx'`
- **根因**: 这两个测试文件在模块级别 import 了依赖外部 SDK 的代码
  - `from src.core.multimodal import ...` → `multimodal.py` 顶层 `import anthropic`
  - `from src.mcp_servers.paper_search import ...` → `paper_search.py` 顶层 `import httpx`
- **修复方案**: 使用 `pytest.importorskip()`
  ```python
  # test_multimodal.py
  anthropic = pytest.importorskip("anthropic", reason="anthropic SDK not installed")
  from src.core.multimodal import ...

  # test_paper_search.py
  httpx = pytest.importorskip("httpx", reason="httpx not installed")
  from src.mcp_servers.paper_search import ...
  ```
- **效果**: 缺少依赖时优雅跳过（2 skipped），不阻塞核心测试
- **教训**:
  1. **全量测试必须在 base 环境也能跑通**——即使某些 SDK 没安装
  2. `pytest.importorskip()` 是比 `try/except` 更地道的 pytest 做法
  3. 之前只单独跑了 `test_knowledge_graph.py`，没有跑全量 `tests/`，这是遗漏

**工程反思**:
- Bug #9 (lazy import in __init__.py) 和 Bug #10 (importorskip) 本质是同一个问题：
  **可选依赖不能成为必选的阻塞点**
- 解决思路也一致：延迟导入 / 条件跳过，让核心模块始终可用

#### 沙箱设计思路

**核心问题**: Agent 生成的仿真代码可能有安全风险（恶意 import、无限循环）

**安全层级设计（4 层防护）**:

| 层级 | 措施 | 防护对象 |
|:---|:---|:---|
| L1 | subprocess 隔离 | Agent 崩溃不影响主进程 |
| L2 | 超时控制 (默认 30s, 最大 120s) | 无限循环 / DoS |
| L3 | 工作目录隔离 (tempdir) | 文件系统污染 |
| L4 | 危险模式正则扫描 (12 个规则) | os.system, eval, rmtree 等 |

**为什么不用 Docker？(ADR-009)**:
- 目标用户是学生，Docker Desktop 增加部署门槛
- 学术仿真代码（numpy/scipy/matplotlib）安全风险低
- L1~L4 对学术场景已足够，Phase 5 保留 Docker 升级路径

**沙箱执行流程**:
```
代码文本 → 安全检查（正则扫描）
    ↓
注入 preamble（matplotlib Agg 后端 + chdir）
    ↓
写入临时文件 exec_XXXX/main.py
    ↓
subprocess.run（超时控制）
    ↓
收集 stdout + stderr + 输出文件（.png/.csv/.json）
    ↓
返回 ExecutionResult（结构化结果 + Markdown 报告）
```

**matplotlib 自动保存的工程细节**:
- 用户代码中的 `plt.show()` 在非交互后端会被忽略
- 在代码末尾自动注入保存脚本：遍历所有 figure → savefig
- 图表保存到执行目录 → output_files 列表返回路径
