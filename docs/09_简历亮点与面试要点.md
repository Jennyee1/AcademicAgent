## 二、关键技术亮点（面试逐条展开）

### 亮点 1: MCP 协议而非 LangChain

**面试话术**:
> "我选择了 MCP 标准协议而非 LangChain 框架。MCP 是 Anthropic 提出的开放标准，
> 类似 HTTP 之于 Web，它解耦了 LLM 和工具层。这意味着我的工具可以被任何兼容 MCP 的
> 客户端调用——Claude Code、VS Code、甚至竞品 LLM——而不被框架锁定。"

**追问准备**:
- Q: "MCP 和 LangChain 的具体区别是什么？"
  - A: MCP 是协议（定义接口），LangChain 是框架（提供实现）。MCP 的 Tool、Resource、Prompt 三类能力对应 LangChain 的 Tool、Retriever、Chain，但 MCP 是跨语言跨厂商的标准。
- Q: "为什么不用 Function Calling？"
  - A: Function Calling 是模型级别的能力（JSON Schema 定义工具），MCP 是系统级别的协议（定义服务发现和调用流程）。我们在 MCP Server 内部仍然使用 Function Calling。

### 亮点 2: Schema-Guided 知识图谱抽取

**面试话术**:
> "知识图谱的核心挑战是 LLM 的输出不可控。我设计了 Schema-Guided 抽取：
> 在 Prompt 中嵌入严格的 Schema 定义（7 种节点 + 11 种关系 + JSON 示例），
> 引导 LLM 输出符合预期的结构化数据。配合三级 JSON 解析容错（直接解析 → 正则提取 → 宽松修复），
> 抽取可靠性从 ~70% 提升到 ~95%。"

**追问准备**:
- Q: "Schema-Guided 和 ReAct 有什么关系？"
  - A: ReAct 决定"做什么"（调用哪个工具），Schema-Guided 决定"输出什么格式"。两者配合：ReAct 选择调用抽取工具 → Schema-Guided 控制抽取输出格式。
- Q: "为什么不用 Structured Output？"
  - A: Claude 的 Structured Output 功能确实可用，但 Schema-Guided Prompting 对模型版本更鲁棒，且支持渐进式降级。

### 亮点 3: PageRank 驱动的学习路径推荐

**面试话术**:
> "学术知识图谱的核心价值是发现'你以为你懂但其实不懂'的概念。
> 我用 PageRank 算法计算每个概念的'基础程度'——被越多方法和论文引用的概念，
> PageRank 越高，就越应该优先学习。再结合三种盲区检测（基础缺失、孤立概念、单源依赖）
> 和拓扑排序确定先后依赖关系，生成加权后的个性化学习路径。"

**追问准备**:
- Q: "PageRank 的阻尼因子是多少？为什么？"
  - A: 0.85，这是 Google 的经典值。语义上表示 85% 概率按引用链传播重要性，15% 概率随机跳转。在学术图谱中，这避免了强连通分量内的重要性震荡。
- Q: "知识图谱有环怎么办？"
  - A: 用 SCC 凝缩（Strongly Connected Components condensation），把环缩成超节点，保证 DAG 后再拓扑排序。这是图论中处理有向图环的标准工业做法。

### 亮点 4: 4 层安全沙箱

**面试话术**:
> "Agent 生成的代码不能直接 exec()，这有命令注入风险。
> 我设计了 4 层安全防护：subprocess 进程隔离、超时控制（防 DoS）、
> 临时工作目录隔离、以及 12 条正则规则的危险模式扫描（os.system、eval、shutil.rmtree 等）。
> MVP 阶段不用 Docker 是有意为之——学术场景下 numpy/matplotlib 代码风险较低，
> 而 Docker 增加了学生用户的部署门槛。"

### 亮点 5: Prompt Engineering as Configuration

**面试话术**:
> "我把所有 Prompt 从代码中提取到独立的 .txt 文件中管理，
> 实现了'Prompt as Configuration'模式。这让 Prompt 调优不需要改代码，
> Git diff 可以精确追踪每次 Prompt 变更，
> 而且同一张图表可以用不同 Prompt 文件做 A/B 测试。"

## 五、STAR 故事模板

### 故事 1: 知识图谱抽取的三级容错

- **S**: LLM 输出 JSON 经常格式不规范（多余逗号、markdown 包裹、嵌套错误）
- **T**: 让抽取可靠性达到 95% 以上
- **A**: 设计三级解析：直接 json.loads → 正则提取代码块 → 宽松修复（去注释、补括号）
- **R**: 从 ~70% 可靠性提升到 ~95%，核心测试 0 failure

### 故事 2: 可选依赖不阻塞核心功能

- **S**: anthropic SDK 未安装时，全量 pytest 报 ImportError
- **T**: 让核心模块测试在任何环境都能跑通
- **A**: 三管齐下——`__init__.py` lazy import、`pytest.importorskip()` 条件跳过、依赖分层（核心/API/工具）
- **R**: 51 → 79 tests pass，0 collection error，CI 在无 API Key 环境也能验证核心逻辑

### 故事 3: 从论文到代码的闭环

- **S**: 学生读论文后想验证结果，但从零写通信仿真代码容易出错
- **T**: 让 Agent 能帮学生快速复现论文方法
- **A**: 预置 3 个通信领域模板（OFDM/MIMO/MUSIC）+ 参数覆盖 API + subprocess 沙箱执行
- **R**: "用 OFDM 模板跑个仿真" → 10 秒内输出 BER 曲线，论文与仿真结果可对照验证

## 六、常见面试问题

### Q: "为什么做这个项目？"
> 我博士方向是通信感知，读论文是日常工作。
> 我发现现有工具（Elicit、Connected Papers）都是通用的、被动的，
> 不能理解通信领域的"信号流图"和"BER 曲线"，也不能帮你发现知识盲区。
> 所以我设计了 ScholarMind——一个能看懂你领域论文、记住你读了什么、
> 告诉你应该学什么、还能帮你跑仿真代码的学术 Agent。

### Q: "最大的技术挑战是什么？"
> LLM 输出的不确定性。知识图谱抽取需要结构化输出，但 LLM 经常输出不规范的 JSON。
> 我通过 Schema-Guided Prompting + 三级解析容错解决了这个问题。
> 另外，可选依赖的管理也是个坑——我花了不少时间让核心模块在没有 API Key 的环境下也能测试通过。

### Q: "如果重新做一遍，你会改什么？"
> 1. 一开始就用 `from __future__ import annotations`，而不是后面逐个文件加
> 2. 一开始就把依赖分为 core/api/tool 三层，而不是后面踩坑才分
> 3. 考虑用 Pydantic 做 Schema 验证，比 dataclass 更适合 LLM 输出解析
