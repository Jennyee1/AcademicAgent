## 1. 快速上手

### 1.1 环境准备

```bash
# 克隆项目
git clone https://github.com/Jennyee1/AcademicAgent.git
cd AcademicAgent

# 安装依赖
pip install -r requirements.txt

# 配置环境变量
cp .env.example .env
# 编辑 .env，填入你的 API Key
```

### 1.2 依赖分层

ScholarMind 的依赖分为三层，避免"装不上就跑不了"的问题：

| 层级 | 依赖 | 用途 | 必需？ |
|:---|:---|:---|:---|
| 核心层 | `networkx`, `python-dotenv` | 图谱存储、配置加载 | ✅ 必须 |
| API 层 | `anthropic`, `httpx` | Claude API、论文搜索 | 🟡 使用时才需要 |
| MCP 层 | `mcp[cli]` | MCP Server 运行 | 🟡 Claude Code 集成时需要 |
| 工具层 | `pymupdf` | PDF 解析 | 🟡 论文分析时需要 |

**不装 anthropic/httpx 也能跑测试**：核心模块（schema, graph_store, graph_analyzer）完全离线可用。

### 1.3 运行测试

```bash
# 运行所有离线测试（推荐首次验证）
python -m pytest tests/ -v

# 运行特定模块的测试
python -m pytest tests/test_knowledge_graph.py -v
python -m pytest tests/test_graph_analyzer.py -v

# 运行集成测试（需要网络 + API Key）
python -m pytest tests/ -v -m integration
```

### 1.4 在 Claude Code 中使用

ScholarMind 设计为 Claude Code 的 MCP 工具集。注册方式：

```bash
# 方式 1: 自动发现（推荐）
# Claude Code 打开项目目录时，会自动读取 .claude/mcp.json

# 方式 2: 手动注册
claude mcp add paper-search -- python src/mcp_servers/paper_search.py
claude mcp add knowledge-graph -- python src/mcp_servers/knowledge_graph.py
claude mcp add learning-path -- python src/mcp_servers/learning_path.py
```

## 3. 各模块详解

### 3.1 Phase 0: 论文搜索 (`paper_search.py`)

**是什么**: Agent 的"信息入口"，从学术数据库搜索论文。

**做什么**:
- `search_papers(query, limit, year_range)` — Semantic Scholar 搜索
- `get_paper_details(paper_id)` — 获取详细元数据
- `get_related_papers(paper_id)` — 获取引用关系
- `search_arxiv(query, limit)` — arXiv 最新预印本搜索

**为什么这样设计**:
- 双源搜索覆盖面更广（Semantic Scholar 有引用数据，arXiv 最新）
- 每篇论文都带 Paper ID / DOI，保证可验证性
- 指数退避重试（3 次），应对 API 限速

**依赖**: `httpx`（异步 HTTP 客户端）

---

### 3.2 Phase 1: 多模态论文理解 (`multimodal.py`)

**是什么**: Agent 的"眼睛"，能看懂论文中的图表。

**做什么**:
- `FigureAnalyzer.analyze(image_b64, context)` — 分析一张图表
- `PaperStructureParser.parse_sections(text)` — 解析论文章节结构
- 自动检测图表类型 → 选择对应分析策略 → 返回结构化 JSON

**为什么这样设计**:
- **Strategy 模式**: 不同图表类型需要不同的 Prompt（框图需要提取组件，曲线需要提取性能数据）
- **两步分析**: 先识别类型（低 token），再深入分析（高质量）
- **Prompt 外置**: 放在 `prompts/` 目录，调优不改代码

**支持的图表类型**:

| 类型 | 说明 | 典型场景 |
|:---|:---|:---|
| block_diagram | 系统框图 | OFDM 收发机、ISAC 系统架构 |
| signal_flow | 信号流图 | 算法流程、信号处理链路 |
| performance_curve | 性能曲线 | BER vs SNR、RMSE 对比 |
| table | 数据表格 | 仿真参数表、性能对比表 |
| equation | 数学公式 | 优化目标函数、信道模型 |
| photo | 实验照片 | 天线阵列、测试场景 |

**依赖**: `anthropic`（Claude Vision API），`pymupdf`（PDF 解析）

---

### 3.3 Phase 2: 知识图谱 (`knowledge/`)

**是什么**: Agent 的"记忆系统"，将分散的论文知识组织成结构化网络。

#### `schema.py` — 领域 Schema 定义

定义了知识图谱中"什么东西可以存"：

**节点类型（7 种）**:
| 类型 | 说明 | 示例 |
|:---|:---|:---|
| Paper | 论文 | "ISAC OFDM Sensing Design" |
| Author | 作者 | "Zhang et al." |
| Concept | 概念 | "OFDM", "MIMO", "Beamforming" |
| Method | 方法 | "UKF Localization", "Beam Tracking" |
| Dataset | 数据集 | "COST 2100 Channel Model" |
| Metric | 评价指标 | "RMSE", "BER", "Spectral Efficiency" |
| Tool | 工具 | "MATLAB", "PyTorch" |

**关系类型（11 种）**:
| 关系 | 含义 | 示例 |
|:---|:---|:---|
| PROPOSES | 提出 | Paper → Method |
| USES | 使用 | Method → Concept |
| IMPROVES | 改进 | Method → Method |
| EXTENDS | 扩展 | Paper → Paper |
| COMPARES_WITH | 对比 | Method → Method |
| CITES | 引用 | Paper → Paper |
| AUTHORED_BY | 作者 | Paper → Author |
| EVALUATED_BY | 使用指标 | Method → Metric |
| BELONGS_TO | 属于 | Concept → Concept |
| RELATED_TO | 关联 | Concept → Concept |
| TESTED_ON | 测试 | Method → Dataset |

**关键设计**: 自动去重 ID 生成（标签标准化 + MD5 hash），确保"OFDM"和"ofdm"映射到同一节点。

#### `graph_store.py` — 图谱存储引擎

**做什么**:
- `add_node(node)` — 添加节点（自动去重合并）
- `add_edge(edge)` — 添加关系
- `query_neighbors(node_id, max_depth)` — BFS 多跳查询
- `search_nodes(keyword)` — 关键词搜索
- `save() / load()` — JSON 持久化
- `get_stats()` — 统计摘要

**持久化格式**: JSON 文件（`data/knowledge_graph.json`），人类可读 + Git 可追踪。

#### `extractor.py` — LLM 知识抽取

**做什么**: 输入论文文本 → 输出结构化的节点和关系。

**工作流程**:
```
论文文本 → 分段（每段 ~3000 字符）
    ↓
每段 + Schema 描述 → Claude API（Schema-guided Prompt）
    ↓
解析 JSON 输出（3 策略 fallback）
    ↓
本地去重合并 → ExtractionResult
```

**依赖**: `anthropic`

### 3.4 Phase 3: 学习路径规划 (`graph_analyzer.py`)

**是什么**: Agent 的"导师"，分析你的知识结构并推荐学习方向。

#### 核心算法 1: 概念重要性评分

```
importance = 0.4 × PageRank + 0.3 × Degree + 0.2 × InDegree + 0.1 × Betweenness
```

- **PageRank**: 越多概念依赖它，它越重要（类似论文引用传递）
- **Degree**: 直接连接越多，知识越"枢纽"
- **InDegree**: 被引用越多，越是"基础知识"
- **Betweenness**: 连接不同知识岛的"桥梁"概念

#### 核心算法 2: 知识盲区检测

| 盲区类型 | 检测逻辑 | 补救建议 |
|:---|:---|:---|
| 🔴 foundation_gap | 核心概念但属性稀疏 | 读教材/综述 |
| 🟡 isolated_concept | 度≤1的非论文节点 | 探索关联关系 |
| 🟠 single_source | 只从1篇论文了解 | 交叉阅读验证 |

#### 核心算法 3: 学习路径生成

```
priority = 0.4 × importance + 0.4 × gap_severity + 0.2 × topo_boost
```

使用 SCC 凝缩处理有向图的环，然后拓扑排序确定先后依赖关系。

## 5. Skills 工作流

Skills 是通过 `.claude/skills/*/SKILL.md` 定义的多步骤工作流。

### 5.1 paper-research

**触发**: "帮我搜论文", "ISAC 领域有什么研究"
**流程**: 理解意图 → 多源搜索 → 智能过滤 → 表格输出

### 5.2 paper-analysis

**触发**: "分析这篇论文", "这篇论文说了什么"
**流程**: 结构扫描 → 核心内容分析 → 知识抽取 → 结构化笔记

### 5.3 knowledge-extract

**触发**: "把这篇论文存到知识图谱", "提取知识"
**流程**: 图谱预检 → LLM 抽取 → 验证结果 → 探索连接

### 5.4 learning-path

**触发**: "我接下来学什么", "规划学习路径"
**流程**: 评估现状 → 检测盲区 → 生成路径 → 行动建议

## 6. 测试体系

### 6.1 测试策略

| 测试文件 | 测试数 | 类型 | 依赖 |
|:---|:---|:---|:---|
| test_knowledge_graph.py | 26 | 单元测试 | 无（核心层） |
| test_graph_analyzer.py | 25 | 单元测试 | 无（核心层） |
| test_multimodal.py | ~10 | 单元测试 | anthropic（自动跳过） |
| test_paper_search.py | ~8 | 集成测试 | httpx（自动跳过） |

**测试分层原则**:
- 核心层测试（schema, graph_store, graph_analyzer）：零外部依赖，永远能跑
- API 层测试（multimodal, paper_search）：依赖缺失时自动 skip，不阻塞 CI
- 集成测试（标记 `@pytest.mark.integration`）：需要网络

### 6.2 运行命令

```bash
# 只跑核心测试（推荐日常开发用）
python -m pytest tests/test_knowledge_graph.py tests/test_graph_analyzer.py -v

# 跑全部（有依赖的自动 skip）
python -m pytest tests/ -v

# 跑集成测试（需要 API Key + 网络）
python -m pytest tests/ -v -m integration
```

## 8. 典型使用场景

### 场景 1: "帮我了解 ISAC 这个领域"

```
你: 帮我搜一下 ISAC 领域最近 3 年的重要论文

Agent 动作:
  1. 调用 search_papers("ISAC", year_range="2023-2026")
  2. 调用 search_arxiv("integrated sensing and communication")
  3. 结果合并去重，按引用数排序
  4. 返回结构化论文表格
```

### 场景 2: "分析这篇论文并存入知识图谱"

```
你: 分析 papers/isac_ofdm_2024.pdf 并存到知识图谱

Agent 动作:
  1. 调用 analyze_pdf 提取章节
  2. 调用 analyze_figure 分析关键图表
  3. 调用 add_paper_to_graph 抽取知识入图
  4. 返回：新增 12 个节点、8 个关系
```

### 场景 3: "我接下来应该学什么？"

```
你: 我已经读了几篇论文了，接下来应该学什么？

Agent 动作:
  1. 调用 get_concept_importance 看核心概念
  2. 调用 detect_gaps 检测盲区
  3. 调用 analyze_knowledge 生成学习路径
  4. 返回：
     - 健康度评估（"🟡 起步阶段"）
     - 3 个知识盲区（Beamforming 属于基础缺失）
     - 推荐路径：1. OFDM → 2. Beamforming → 3. Channel Estimation
```

## 9. FAQ

### Q: 没有 Anthropic API Key 能用吗？
**A**: 可以！论文搜索（Semantic Scholar）不需要 API Key（有限速），知识图谱的创建/查询/分析都是离线的。只有 LLM 抽取和图表分析需要 API Key。

### Q: 知识图谱存在哪里？
**A**: `data/knowledge_graph.json`，一个 JSON 文件。人类可以直接阅读、Git 可以追踪变更。未来可能迁移到 Neo4j。

### Q: 为什么不用 LangChain？
**A**: 详见 [07_技术栈全景与架构决策.md](./07_技术栈全景与架构决策.md) 中的 ADR-006。简短回答：MCP 是标准协议，LangChain 是框架锁定。我们选择"协议 > 框架"。

### Q: Prompt 模板怎么调优？
**A**: 直接编辑 `prompts/domain_prompts/*.txt` 文件，不需要改代码。重启 MCP Server 即可生效。

### Q: 测试为什么有 2 个 skip？
**A**: `test_multimodal.py` 和 `test_paper_search.py` 依赖 `anthropic` 和 `httpx`，在 base conda 环境中未安装。使用 `pytest.importorskip()` 优雅跳过，不影响核心模块测试。
